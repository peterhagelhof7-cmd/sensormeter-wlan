Implementierungsplan

Reihenfolge von der Projektstruktur bis zur vollständigen Firmware gemäß lastenheft.txt / pflichtenheft.txt. Kein bestehender Prototyp — Startpunkt ist ein leeres PlatformIO-Projekt. STUFE 1 Grundgerüst & Zustandsmodell
P0

Projektstruktur & Boot-Zustandsautomat

DataManager · ConfigManager · NetworkManager · TimeManager · StorageManager (Gerüst)
  • PlatformIO-Projekt (Board esp32dev), Pin-Definitionen zentral (I2C GPIO21/22, DHT22 GPIO4)
  • Zustandsmodell: Boot → Init → WLAN Check → Run / Fallback (kein LAN-Zweig)
  • DataManager mutex-geschützt von Anfang an (Vorbild: Sensormeter-Projekt)
Voraussetzung: — Vereinfacht (kein LAN)
STUFE 2 Netzwerk (WLAN) & Zeit
P1

WLAN-Verbindung, Fallback-AP, NTP

NetworkManager · TimeManager
  • WLAN-Verbindungsaufbau mit gespeicherten Zugangsdaten (DHCP/static)
  • Fallback Access Point "installer"/"installer", wenn Verbindung nach 5 Min. fehlschlägt
  • NTP-Sync (de.pool.ntp.org, CET/CEST), 60s nach Boot, alle 5h
Voraussetzung: P0 Vereinfacht (kein LAN-Vorrang)
STUFE 3 Konfigpersistenz
P2

config.xml auf LittleFS

ConfigManager
  • Laden/Speichern mit Default-Fallback
  • XML-Import/-Export (kein Sensor-2-, kein LAN-Abschnitt)
Voraussetzung: P0 Kernfunktion
STUFE 4 Sensorik
P3

DHT22 (genau ein Sensor)

SensorManager
  • Abfrage alle 60s, Plausibilitätsprüfung
  • Stündlicher 7-Tage-Ringpuffer (168 Werte)
  • Kein Sensor-2-Zweig, keine RJ45-Verdrahtung
Voraussetzung: P2 Vereinfacht (1 statt bis zu 2 Sensoren)
STUFE 5 Display
P4

OLED SSD1306 (I2C, Standardpins)

DisplayManager
  • Boot-Screen mit Countdown
  • Rotierende Seiten (Systemname/WLAN-IP/Uhrzeit/Sensorwert/Status), 10s-Takt
Voraussetzung: P1, P3 Baut auf P1, P3
STUFE 6 Webserver (inkl. lokales OTA-Update)
P5

Hauptseite, Einstellungsseite, OTA

WebServerManager · OtaManager
  • Hauptseite: Systemwerte, Chart.js-Graph, Syslog-Tabelle, values.csv-Download
  • Passwortgeschützte Einstellungsseite: Systemname, WLAN-Scan+Auswahl, Syslog-Server, XML-Import/-Export
  • Lokales OTA per .bin-Upload - Standard-Partitionsschema reicht (ota_0/ota_1 bereits vorhanden), keine eigene partitions.csv nötig
  • Kein HTTPS-Client/Remote-Versionscheck (Flash-Kosten, siehe Sensormeter-Projekt-Begründung)
Voraussetzung: P2, P3 Vereinfacht (kein Sensor-2-Formular)
STUFE 7 SNMP v1
P6

SNMP-Agent (read-only)

SNMPManager
  • Feste OID-Struktur unter .1.3.6.1.4.1.99999.x (System, Netzwerk, Sensor 1, Systemstatus)
  • Bewusst gleiche Basis-OID wie das Sensormeter-Projekt - Sensormeter-Display kann beide Varianten ohne Codeänderung abfragen
  • Sensor-2-Zweig (.4) bleibt unbeantwortet statt belegt
Voraussetzung: P3 Kernfunktion
STUFE 8 Syslog
P7

Syslog-Versand (UDP 514)

SyslogManager
  • Statusreport bei jedem Sensorzyklus
  • Fehler-Events (Sensor/WLAN/NTP) sofort
  • Deaktiviert, solange kein Syslog-Server konfiguriert ist
Voraussetzung: P2, P3 Kernfunktion

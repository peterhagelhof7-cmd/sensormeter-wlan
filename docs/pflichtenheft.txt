
PFLICHTENHEFT – SENSORMETER WLAN (ESP32-WROOM-32 DevKit)

1. SYSTEMARCHITEKTUR

Das System basiert auf einem generischen ESP32-WROOM-32-DevKit (reines
WLAN, kein Ethernet-PHY) mit folgenden Software-Komponenten:

- FreeRTOS (ESP32 Standard)
- AsyncWebServer (Web UI)
- SNMP v1 Stack (Read Only)
- Syslog Client (UDP 514)
- NTP Client
- LittleFS Dateisystem
- Arduino Framework (empfohlen)

Architekturstil wie beim Sensormeter-Projekt (WT32-ETH01): Manager-Klassen
in einem kooperativen Hauptloop, zentrale Datenhaltung über einen
mutex-geschützten DataManager (Thread-Sicherheit siehe dortiges Vorbild –
async Webserver + Sensor-Polling greifen aus unterschiedlichen Kontexten
zu).

------------------------------------------------------------

2. SOFTWARESTRUKTUR

2.1 Tasks (FreeRTOS)

Task: SensorTask
- Intervall: 60 Sekunden
- Funktion:
  - DHT22-Werte lesen (Temperatur, Luftfeuchte)
  - Plausibilitätsprüfung
  - Übergabe an DataManager
  - Trigger: Syslog Sendung

Task: NetworkTask
- Verwaltung WLAN (kein LAN/Ethernet-Zweig, im Unterschied zum
  Sensormeter-Projekt)
- DHCP / Static IP Handling
- Fallback-AP-Aktivierung bei Verbindungsfehler

Task: NTPTask
- Initial nach 60s
- danach alle 5h
- Sync via de.pool.ntp.org
- Fehlerhandling: Zeit als "nicht synchronisiert" markiert, kein
  DHCP-Failover nötig (kein zweites Interface vorhanden)

Task: WebServerTask
- Async HTTP Server
- REST API + HTML UI
- Chart.js Datenbereitstellung

Task: DisplayTask
- OLED Update alle 10 Sekunden
- Rotation der Informationsseiten
- BOOT-Taster (GPIO0) zusätzlich als Eingabe ausgewertet: kurzer Tipp
  schaltet manuell zur nächsten Seite, langes Halten (3s + 20s Countdown,
  Auslösung erst beim Loslassen als Fail-Safe gegen einen verklemmten
  Taster) löst einen Werksreset der Einstellungen aus

Task: SyslogTask
- UDP Versand bei jedem Sensorzyklus
- Fehler-Events sofort senden

Task: MqttTask
- Nur aktiv, wenn in config.xml <mqtt enabled="true"> und Broker-Adresse
  gesetzt (siehe Abschnitt 3.7)
- Verbindungsaufbau throttled (alle 5s ein Versuch, kein Busy-Loop)
- Home-Assistant-Discovery-Payloads (retained) nach jedem
  (Wieder-)Verbindungsaufbau
- Statuswerte bei jedem Sensorzyklus (gleiches Erkennungsmuster wie
  SyslogTask: Aenderung von lastReadMillis, kein eigener Timer)

3. SOFTWAREMODULE

3.1 DataManager
Zentrale Datenhaltung (mutex-geschützt, siehe Sensormeter-Projekt/
DataManager als Vorbild):
- Sensorwert (genau ein Sensor)
- Systemstatus
- Netzwerkstatus (WLAN)
- Zeitstempel
- Ringbuffer für 7 Tage Stundenwerte

3.2 ConfigManager
- Laden config.xml aus LittleFS
- Speichern config.xml
- XML Parser
- Default-Werte fallback

3.3 NetworkManager
- WLAN Init
- Fallback-AP "installer" bei Verbindungsfehler (kein zweites Interface,
  daher einfacher als beim Sensormeter-Projekt: kein LAN/WLAN-Vorrang,
  keine Interface-Auswahl)
- DHCP/static Umschaltung

3.4 TimeManager
- NTP Sync
- Sommer-/Winterzeit (CET/CEST)
- Uptime tracking

3.5 StorageManager
- LittleFS Zugriff
- values.csv write
- JSON-Puffer für Graph

3.6 SensorManager
- Ein DHT22 (kein Sensor-2-Zweig, kein RJ45)

3.7 MqttManager
- Home-Assistant-Anbindung per MQTT-Discovery (PubSubClient), siehe
  docs/lastenheft.txt Abschnitt 14 sowie
  sensormeter-poe/repo/docs/lastenheft.txt Abschnitt 16 fuer das
  vollstaendige Feature-Design (dort inkl. Relais/Aktor-Rolle, hier nur
  Sensor-Rolle, da kein RJ45-Modularanschluss vorhanden)
- topicPrefix() nutzt dieselbe Sanitisierung wie der mDNS-Hostname
  (NetworkManager::sanitizeHostname) statt eines eigenen gespeicherten
  Feldes
- Reconnect throttled (5s-Intervall), Discovery wird nach jedem
  (Wieder-)Verbindungsaufbau neu gesendet (kein Persistenz-Aufwand fuer
  "schon gesendet" ueber einen Reconnect hinweg noetig)

3.8 BrandingManager
- Anbieter-Branding (Weisslabel), siehe docs/lastenheft.txt Abschnitt 15
- Vendor-Name in config.xml (analog anderen Textfeldern), Logo-Binaerdaten
  bewusst NICHT dort, sondern als eigene Rohdatei auf LittleFS
  (/branding-logo.bin, 1024 Byte fest: 128x64 1bpp)
- hasLogo() haelt den Vorhanden-Zustand in einem RAM-Cache statt bei jeder
  Abfrage LittleFS.exists() aufzurufen - diese Bibliotheksfunktion ist
  intern als open(path,"r") implementiert, was bei jedem Fehlschlag eine
  ESP-IDF-VFS-Fehlerzeile im Log erzeugt (siehe docs/entscheidungen.md);
  bei einer alle 10s (Seitenwechsel-Takt) wiederholten Pruefung waere das
  eine dauerhaft wiederkehrende Logzeile gewesen
- Upload-Streaming (beginLogoUpload/writeLogoUploadChunk/endLogoUpload)
  identisches Tmp-Datei-plus-Umbenennen-Muster wie ConfigManager::save()
- Web-Auslieferung ueber WebServerManager: GET /branding/logo.bmp
  synthetisiert on-the-fly einen minimalen 1-Bit-BMP aus dem Rohformat
  (kein PNG/JPEG-Decoder eingebunden)

4. DATENSPEICHERUNG

4.1 Ringbuffer (7 Tage)
- 168 Stundenwerte
- Struktur:

struct HourValue {
  timestamp;
  temperature;
  humidity;
}

4.2 values.csv
- Logging aller Stundenwerte
- Export für Web UI

4.3 config.xml
- Systemkonfiguration
- WLAN-Parameter (kein LAN-Abschnitt)
- MQTT-Parameter (enabled/server/port/user/password) - Topic-Praefix wird
  NICHT gespeichert, siehe 3.7
- Branding-Parameter (vendorName) - Logo-Binaerdaten NICHT hier, siehe 4.4

4.4 branding-logo.bin
- Optionale Datei, nur vorhanden wenn ein Logo hochgeladen wurde
- Rohformat: 128x64 Pixel, 1 Bit pro Pixel, MSB-zuerst je Zeile, exakt
  1024 Byte - siehe docs/lastenheft.txt Abschnitt 15.3

6. FEHLERBEHANDLUNG

6.1 Netzwerkfehler
- WLAN reconnect
- Fallback Access Point "installer"
- DHCP recovery

6.2 NTP Fehler
- retry nach 5 min
- Zeit bleibt bis dahin als "nicht synchronisiert" markiert (kein
  Interface-Failover, da nur WLAN vorhanden)

6.3 Sensorfehler
- syslog event
- letzten gültigen Wert markiert beibehalten statt Absturz

------------------------------------------------------------

7. SNMP IMPLEMENTATION

- SNMP v1 Agent
- read-only community konfigurierbar
- polling optimiert (kein Dauer-Refresh)
- gleiche Basis-OID wie das Sensormeter-Projekt (.1.3.6.1.4.1.99999),
  Sensor-2-Zweig (.4) bleibt unbelegt/unbeantwortet - Kompatibilität mit
  dem SNMP-Client des Sensormeter-Display-Projekts ohne Anpassung dort

8. PERFORMANCE DESIGN

Zielwerte (wie Sensormeter-Projekt):
- CPU load < 40%
- RAM usage < 60%
- Sensor polling minimal (60s)
- Webserver non-blocking (Async)

9. MEMORY LAYOUT

Flash: Standard-Partitionsschema von PlatformIOs `esp32dev`-Board
(unveraendert, KEINE eigene partitions.csv noetig) - verifiziert am
tatsaechlichen Build des Sensormeter-Projekts (WT32-ETH01, identisches
Board-Target, keine eigene Partitionstabelle in dessen platformio.ini):

  nvs       20K
  otadata   8K
  app0 (ota_0)   1280K
  app1 (ota_1)   1280K
  spiffs (LittleFS)  1408K
  coredump  64K

Enthaelt bereits zwei OTA-faehige App-Partitionen (ota_0/ota_1) - lokales
OTA-Update (Abschnitt 6) ist damit ohne zusaetzlichen Partitionsaufwand
umsetzbar, siehe entscheidungen.md.

- Firmware: ~1.0-1.3 MB erwartet (etwas kleiner als das Sensormeter-Projekt,
  da kein Ethernet-Treiber (LAN8720/ETH.h) gelinkt werden muss) - passt
  komfortabel in die 1280K je App-Slot
- LittleFS: 1408K verfuegbar, values.csv (7 Tage Stundenwerte) braucht
  davon nur einen Bruchteil

RAM:
- Heap reserve: >150 kB minimum

------------------------------------------------------------

10. STARTUP FLOW

BOOT →
  Init Hardware →
  Load Config →
  Init WLAN →
  Start Services →
  Wait 60s →
  NTP Sync →
  Normal Operation

Fallback:
  Access Point "installer" →
  Einstellungsseite zur WLAN-Auswahl →
  Reboot nach Speichern

11. ERWEITERBARKEIT

- Bewusst NICHT für einen zweiten Sensor oder Modul-/RJ45-Erweiterung
  vorgesehen (Abgrenzung zum Sensormeter/Sensormeter-PRO-Projekt, das
  genau das bereits abdeckt) - vermeidet doppelten Funktionsumfang
  zwischen den beiden Produktlinien
- MQTT/Home-Assistant-Anbindung umgesetzt (Abschnitt 2 MqttTask, 3.7
  MqttManager, docs/lastenheft.txt Abschnitt 14) - Sensor-Rolle, kein
  Relais/Aktor, siehe docs/entscheidungen.md fuer die
  Ressourcenkosten-Messung
- Cloud-Integration darüber hinaus bewusst ausgeschlossen
